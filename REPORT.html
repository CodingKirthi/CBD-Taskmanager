AI-Assisted Test Development Report

Application: Task Manager REST API (Spring Boot 3.2)

Submission Type: Individual  |  Weighting: 20%

Application Under Test

The application is a Task Manager REST API built with Spring Boot 3.2.3. It exposes endpoints for managing Projects and Tasks and enforces the following business rules:

  • A task cannot have a due date in the past.
  • A completed task (DONE) cannot be reopened to OPEN.
  • A project cannot be deleted while it has IN_PROGRESS tasks.
  • Project names must be unique.

Technology stack: Spring Boot 3.2.3, Spring Data JPA, H2 (file at runtime / in-memory for tests), Jakarta Bean Validation, JUnit 5, Mockito, AssertJ.

Section 1 – Test Strategy

Alignment with the Test Pyramid

The test suite is intentionally structured in three tiers that mirror the Test Pyramid: the cheapest, fastest tests dominate at the base; the fewest, most comprehensive tests sit at the apex.

         /\
        /  \      E2E (4 tests)  – Full HTTP stack, real H2 DB
       /----\
      / Int  \    Integration (9 tests)  – JPA slice + Web slice
     /--------\
    /  Unit    \  Unit (15 tests)  – No Spring, Mockito only
   /____________\

Unit Tests (15 tests, ~0.7 s total)

Unit tests dominate the suite (54%). They test business logic in isolation using @ExtendWith(MockitoExtension.class) – no Spring context is ever loaded. All dependencies are replaced with @Mock stubs; the class under test is wired via @InjectMocks.

Integration Tests (9 tests, ~3 s total)

Integration tests operate at two Spring slices:

  • @DataJpaTest (4 tests) – loads only JPA, repositories, and H2. Tests derived query methods against a real schema.
  • @WebMvcTest (5 tests) – loads only the web layer. The service is replaced with @MockBean. Tests HTTP binding, JSON serialisation, and bean validation errors handled by GlobalExceptionHandler.

End-to-End Tests (4 tests, ~11 s total)

E2E tests use @SpringBootTest(webEnvironment = RANDOM_PORT) + TestRestTemplate. The full application boots against an H2 in-memory database. These tests exercise the complete request → service → repository → response pipeline.

Section 2 – Dependency Injection and Mocking Approach

How Constructor Injection Enables Testability

All service classes use constructor injection exclusively:

@Service
public class TaskService {
    private final TaskRepository taskRepository;
    private final ProjectRepository projectRepository;
    private final TaskMapper taskMapper;

    public TaskService(TaskRepository taskRepository,
                       ProjectRepository projectRepository,
                       TaskMapper taskMapper) {
        this.taskRepository = taskRepository;
        this.projectRepository = projectRepository;
        this.taskMapper = taskMapper;
    }
}

Because dependencies are declared in the constructor, a unit test can instantiate TaskService directly by passing mock objects. Mockito's @InjectMocks exploits this same constructor to inject @Mock instances. No Spring container is needed.

Why not field injection? Field injection (@Autowired TaskRepository repo;) would make this impossible – a test would need to start the full Spring context or use brittle reflection hacks to inject mocks.

Where Mocks Were Used and Why

Test ClassWhat is MockedWhy
TaskServiceTest TaskRepository, ProjectRepository, TaskMapper Isolate TaskService business logic from persistence. Control exact return values (e.g. Optional.empty() to trigger 404 logic).
ProjectServiceTest ProjectRepository, TaskRepository, ProjectMapper Test delete-blocking logic without a real database.
TaskControllerWebMvcTest TaskService (@MockBean) Isolate the HTTP layer. Test controller annotations, request validation, and GlobalExceptionHandler responses independently of service logic.

ArgumentCaptor was used in TaskServiceTest to inspect the Task entity passed to taskRepository.save(), confirming completedAt was set before the save call – a side-effect that cannot be observed from the return value alone.

Where Mocks Were Avoided and Why

Test ClassMock AvoidedWhy
TaskMapperTest None – no dependencies TaskMapper is a pure function with no collaborators. Direct instantiation. Zero framework overhead.
TaskRepositoryIntegrationTest No mocks The purpose is to verify that derived query methods produce correct SQL. Mocking the repository would defeat this entirely.
TaskManagerE2ETest No mocks The purpose is to verify that all layers work together correctly. Mocking any layer would undermine this guarantee.

Section 4 – AI Interaction Log

Example 1: AI Output Accepted As-Is

Prompt given to AI:

"Write a @DataJpaTest integration test for TaskRepository. Test the findByPriorityAndStatusNot query method. Persist test data using TestEntityManager, clear the entity manager cache, then assert that only tasks with CRITICAL priority and non-DONE status are returned. Use AssertJ assertions."

AI-generated output (accepted without modification):

@Test
void findByPriorityAndStatusNot_returnsCriticalNonDoneTasks() {
    Project project = entityManager.persistAndFlush(new Project("Critical Project", null));

    entityManager.persistAndFlush(new Task("Critical open", null, TaskStatus.OPEN,
            TaskPriority.CRITICAL, null, project));
    entityManager.persistAndFlush(new Task("Critical done", null, TaskStatus.DONE,
            TaskPriority.CRITICAL, null, project));
    entityManager.persistAndFlush(new Task("Medium open", null, TaskStatus.OPEN,
            TaskPriority.MEDIUM, null, project));

    entityManager.clear();

    List<Task> criticalNonDone = taskRepository.findByPriorityAndStatusNot(
            TaskPriority.CRITICAL, TaskStatus.DONE);

    assertThat(criticalNonDone).hasSize(1);
    assertThat(criticalNonDone.get(0).getTitle()).isEqualTo("Critical open");
}

Justification:
The AI correctly used persistAndFlush() followed by entityManager.clear() – the proper @DataJpaTest pattern: flush to write to H2, then clear the first-level cache to force a real SQL SELECT. Without clear(), Hibernate returns cached objects rather than executing the query, potentially masking bugs. The test covers exactly one scenario, has a clean Arrange-Act-Assert structure, and uses typed AssertJ assertions.

Example 2: AI Output Modified

Original prompt:

"Write a unit test for TaskService.updateTaskStatus that verifies the completedAt timestamp is set when a task is transitioned to DONE."

Original AI-generated output:

@Test
void updateTaskStatus_toDone_setsCompletedAt() {
    Task task = new Task();
    task.setId(5L);
    task.setStatus(TaskStatus.IN_PROGRESS);

    when(taskRepository.findById(5L)).thenReturn(Optional.of(task));
    when(taskRepository.save(task)).thenReturn(task);
    when(taskMapper.toDTO(task)).thenReturn(new TaskDTO());

    taskService.updateTaskStatus(5L, TaskStatus.DONE);

    assertThat(task.getCompletedAt()).isNotNull();  // <-- asserts on local variable
}

Problem with the original:
The test asserted on the local task variable, relying on Java reference identity rather than verifying the entity passed to save(). If the service were refactored to copy the entity before saving, the assertion would pass on the original object even though the saved copy had a null completedAt. Additionally, the no-arg new Task() leaves the project field null, which would throw a NullPointerException when TaskMapper.toDTO() calls task.getProject().getName().

Final modified version:

@Test
void updateTaskStatus_toDone_setsCompletedAtTimestamp() {
    Project project = new Project("Gamma", "Gamma project");
    project.setId(3L);

    Task task = new Task("Fix bug", null, TaskStatus.IN_PROGRESS,
            TaskPriority.HIGH, null, project);
    task.setId(5L);

    when(taskRepository.findById(5L)).thenReturn(Optional.of(task));
    when(taskRepository.save(any(Task.class))).thenAnswer(inv -> inv.getArgument(0));
    when(taskMapper.toDTO(any(Task.class))).thenReturn(
            new TaskDTO(5L, "Fix bug", null, TaskStatus.DONE, TaskPriority.HIGH,
                        null, LocalDateTime.now(), 3L, "Gamma"));

    taskService.updateTaskStatus(5L, TaskStatus.DONE);

    ArgumentCaptor<Task> captor = ArgumentCaptor.forClass(Task.class);
    verify(taskRepository).save(captor.capture());
    assertThat(captor.getValue().getCompletedAt()).isNotNull();
    assertThat(captor.getValue().getStatus()).isEqualTo(TaskStatus.DONE);
}

What changed and why:

  1. ArgumentCaptor was introduced to capture the entity actually passed to save(), testing the correct contract: that the service sets completedAt before persisting.
  2. any(Task.class) replaces the specific reference match so the stub works if the service copies the entity.
  3. A real Project was added to prevent the NullPointerException in the mapper.

Example 3: AI Output Rejected

Prompt:

"Write a unit test for ProjectService.deleteProject that verifies the method throws BusinessRuleException when the project has in-progress tasks."

Rejected AI-generated output (excerpt):

@SpringBootTest
@Transactional
class ProjectServiceDeleteTest {

    @Autowired
    private ProjectService projectService;

    @Autowired
    private ProjectRepository projectRepository;

    @Autowired
    private TaskRepository taskRepository;

    @Test
    void deleteProject_withInProgressTasks_throwsException() {
        Project project = new Project("Doomed Project", "desc");
        projectRepository.save(project);

        Task task = new Task("Active work", null, TaskStatus.IN_PROGRESS, ...);
        taskRepository.save(task);

        assertThrows(BusinessRuleException.class,
                () -> projectService.deleteProject(project.getId()));
    }
}

Technical justification for rejection:

  1. Wrong layer of the Test Pyramid. The test uses @SpringBootTest, which boots the entire application. The same business rule can be verified in under 1 ms using Mockito; this test takes over 5 seconds and carries JPA and HTTP startup overhead.
  2. Misuse of the Spring context for a unit concern. The business rule is entirely contained within ProjectService.deleteProject(). Testing it through the full context does not provide additional confidence – it only adds startup cost and coupling to infrastructure.
  3. Fragile implicit contract. The test relies on @Autowired repositories, H2 schema creation, and @Transactional rollback. If any of these change (e.g., the test profile is misconfigured), the test fails for reasons unrelated to the business rule.

The correct unit test – already present in ProjectServiceTest – achieves the same result in <1 ms with zero infrastructure:

@Test
void deleteProject_whenHasInProgressTasks_throwsBusinessRuleException() {
    when(projectRepository.existsById(10L)).thenReturn(true);
    when(taskRepository.countByProjectIdAndStatus(10L, TaskStatus.IN_PROGRESS)).thenReturn(2L);

    assertThatThrownBy(() -> projectService.deleteProject(10L))
            .isInstanceOf(BusinessRuleException.class)
            .hasMessageContaining("in-progress");

    verify(projectRepository, never()).deleteById(any());
}

Example 4: AI-Identified Gap or Improvement

Analysis prompt:

"Review the unit test suite for TaskService. Identify any missing test scenarios or edge cases not currently covered."

AI's suggestion:

"The current suite does not test deleteTask when the task does not exist. TaskService.deleteTask calls taskRepository.existsById(id) and throws ResourceNotFoundException if it returns false. The only existing deleteTask test covers the happy path. A test for the not-found case would verify the guard clause and that deleteById is never called."

Action taken – test added:

@Test
void deleteTask_whenTaskDoesNotExist_throwsResourceNotFoundException() {
    when(taskRepository.existsById(404L)).thenReturn(false);

    assertThatThrownBy(() -> taskService.deleteTask(404L))
            .isInstanceOf(ResourceNotFoundException.class)
            .hasMessageContaining("404");

    verify(taskRepository, never()).deleteById(any());
}

The AI correctly identified that every guard clause deserves both a passing and a failing test. The suggestion was accepted and incorporated, bringing the unit test count for TaskServiceTest to 8 tests.

Example 5: AI Output Constrained by Explicit Rules

Prompt with explicit constraints:

"Write a unit test for TaskMapper.toDTO. Apply the following constraints:
  1. Do NOT use @ExtendWith(MockitoExtension.class) or any Mockito annotations.
  2. Do NOT start a Spring context (@SpringBootTest, @DataJpaTest, @WebMvcTest).
  3. Instantiate TaskMapper with new TaskMapper() directly.
  4. Use only JUnit 5 and AssertJ – no Spring test utilities.
The test should verify that all fields of the Task entity are correctly mapped to TaskDTO."

AI-generated output produced under this constraint:

class TaskMapperTest {

    private final TaskMapper mapper = new TaskMapper();

    @Test
    void toDTO_mapsAllFieldsCorrectly() {
        Project project = new Project("Mapper Project", "desc");
        project.setId(5L);

        Task task = new Task("Map me", "description here",
                TaskStatus.IN_PROGRESS, TaskPriority.HIGH,
                LocalDate.of(2026, 6, 1), project);
        task.setId(99L);
        task.setCompletedAt(LocalDateTime.of(2026, 5, 20, 10, 0));

        TaskDTO dto = mapper.toDTO(task);

        assertThat(dto.getId()).isEqualTo(99L);
        assertThat(dto.getTitle()).isEqualTo("Map me");
        assertThat(dto.getDescription()).isEqualTo("description here");
        assertThat(dto.getStatus()).isEqualTo(TaskStatus.IN_PROGRESS);
        assertThat(dto.getPriority()).isEqualTo(TaskPriority.HIGH);
        assertThat(dto.getDueDate()).isEqualTo(LocalDate.of(2026, 6, 1));
        assertThat(dto.getCompletedAt()).isEqualTo(LocalDateTime.of(2026, 5, 20, 10, 0));
        assertThat(dto.getProjectId()).isEqualTo(5L);
        assertThat(dto.getProjectName()).isEqualTo("Mapper Project");
    }
}

Explanation:

Behaviour being prevented: Without the constraint, AI tools default to adding @ExtendWith(MockitoExtension.class) or @SpringBootTest even for the simplest test classes. For TaskMapper – which has no dependencies and performs only field copies – this creates unnecessary overhead.

Why the constraint was necessary: TaskMapper is stateless and dependency-free. Testing it should cost nothing more than instantiating it and calling a method. Mockito's extension performs bytecode manipulation; Spring's context performs component scanning, JPA schema creation, and bean wiring. None of this is needed here.

How the constraint improved test quality: The resulting test runs in under 1 millisecond with zero framework overhead. It tests exactly the public contract of the class (field mapping) and nothing else. It cannot be broken by database configuration, Spring Boot version changes, or profile settings – a textbook example of a correctly placed test at the very base of the Test Pyramid.

Section 5 – Evaluation and Reflection

Strengths of AI-Generated Tests

Strength 1: Boilerplate generation speed.
AI generated the structural boilerplate of test classes – imports, class-level annotations, mock field declarations, and @BeforeEach setup – in seconds. For TaskRepositoryIntegrationTest, the AI correctly identified @DataJpaTest, @ActiveProfiles("test"), TestEntityManager, and the persistAndFlush/clear pattern without being prompted. Writing this setup manually is error-prone; the AI produced it correctly on the first attempt.

Strength 2: Comprehensive boundary condition identification.
When asked to review the test suite (Example 4), the AI correctly identified the missing deleteTask not-found test. It systematically recognised that every guard clause requires both a passing and a failing test – the kind of thorough enumeration humans focusing on the happy path sometimes miss.

Weaknesses of AI-Generated Tests

Weakness 1: Default to heavy test infrastructure.
Example 3 demonstrates the most consistent AI failure: reaching for @SpringBootTest regardless of whether the scenario requires it. The AI generated a full-context integration test for a scenario needing only two mock objects. Left unconstrained, AI will load thousands of lines of Spring infrastructure for a test that should run in milliseconds.

Weakness 2: Testing implementation details rather than contracts.
In Example 2, the original AI-generated test asserted on the local task variable rather than using ArgumentCaptor to inspect the entity passed to save(). The test passed for the wrong reason: it verified Java reference semantics, not that the service sets completedAt before persisting. A test that passes for the wrong reason can survive a refactoring that breaks the actual contract.

Observed AI Failure Mode: Incorrect Assumption About Test Scope

Across multiple interactions, AI tools consistently pattern-match "Spring Boot application" → "use @SpringBootTest" without reasoning about whether the specific test requires the full context. The practical consequence is test suites that are slow, brittle, and fragile under configuration changes – for reasons entirely unrelated to the business logic under test. This is not occasional; it is the default behaviour unless explicitly constrained (as demonstrated in Example 5).

Example Where Human Judgement Improved Test Quality

In Example 2, the AI's assertion on the mutated local variable appeared correct at first glance. It was human review of the reason for the assertion – not just the assertion itself – that identified the problem. The human question was: "Is this test actually verifying that the entity passed to save() has completedAt set, or is it just verifying that the service mutates the same object it received?" AI does not naturally ask this kind of reflective question about its own output; it generates locally plausible code without considering whether the assertion survives reasonable refactoring. The fix – introducing ArgumentCaptor – came from applying the testing principle that assertions should verify observable outputs through public interfaces, not through internal object state.

One Improvement to the AI-Assisted Testing Approach

The most valuable single improvement would be to establish a constraint prompt template prepended to every test-generation request. Based on patterns observed in this assignment, the template would include:

Rules for all tests in this project:
1. If the class under test has no Spring dependencies, use @ExtendWith(MockitoExtension.class) only.
2. Never use @SpringBootTest for a test that can be written with @WebMvcTest, @DataJpaTest,
   or plain Mockito.
3. When asserting on side-effects (calls to save(), delete()), use ArgumentCaptor to verify
   the exact argument passed, not the local variable.
4. Every method with a guard clause requires at least two tests: one where the guard passes,
   one where it throws.

Rather than correcting AI output after the fact, this approach constrains it at the source – eliminating the most frequently observed failure modes before any code is generated.
